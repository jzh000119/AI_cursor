# 0001 — 基于标签的 Ticket 管理工具：需求与设计文档

---

## 1. 项目背景与目标

### 1.1 背景

在日常工作中，常需要对任务/事项进行记录、分类和跟踪。本工具提供一个轻量级的 Ticket 管理方案，以「标签」作为核心分类手段，无需用户账户系统，面向本地或小团队内部使用。

### 1.2 目标

- 提供简单易用的 Ticket 创建、编辑、删除与完成功能。
- 以标签为核心对 Ticket 进行多维度分类。
- 支持按标签筛选与按标题关键词搜索，快速定位目标 Ticket。
- 前后端分离，后端提供 REST API，前端提供现代化单页应用。

### 1.3 术语定义

| 术语 | 含义 |
|------|------|
| Ticket | 一条任务/事项条目，包含标题、描述、颜色、状态、关联标签等信息 |
| 标签（Tag） | 用于分类的标记，一个标签可关联多个 Ticket，一个 Ticket 可挂多个标签（多对多） |
| 完成状态 | Ticket 的「未完成」或「已完成」两种状态之一 |
| 颜色（Color） | 为 Ticket 指定的视觉颜色标识，从预设调色板中选取，用于快速区分不同 Ticket |

## 3. 功能需求

### 3.1 Ticket 管理

#### 3.1.1 创建 Ticket

- 用户可新建一条 Ticket。
- **必填字段**：标题（title），长度 1–200 个字符，不可为空白字符。
- **可选字段**：描述（description，纯文本）；颜色（color）；初始关联的标签列表。
- 创建成功后，状态默认为「未完成」，颜色默认为「无颜色」（null）。
- 系统自动记录创建时间（created_at）与最后更新时间（updated_at）。
- 标题校验失败时，前端表单内显示具体错误提示，不提交请求。

#### 3.1.2 编辑 Ticket

- 用户可修改已有 Ticket 的标题、描述与颜色。
- 标签的增减通过「标签关联」功能完成（见 3.2），不在编辑表单中处理。
- 标题校验规则同「创建」。
- 颜色可设置为预设色板中的某个颜色，也可置空（移除颜色标识）。
- 编辑成功后，updated_at 更新为当前时间。

#### 3.1.3 删除 Ticket

- 用户可删除一条 Ticket。
- 删除前，前端弹出确认对话框，用户确认后方可提交删除请求。
- 删除后，该 Ticket 与所有标签的关联记录一并删除（数据库级联处理）。
- 删除成功后，列表自动刷新。

#### 3.1.4 完成 / 取消完成 Ticket

- 每条 Ticket 具有「未完成」（completed = false）与「已完成」（completed = true）两种状态。
- 用户可一键切换状态，操作即时生效（请求后更新 UI，无需整页刷新）。
- 列表中已完成 Ticket 以视觉上可区分的样式展示（如标题加删除线、行背景变灰）。

---

### 3.2 标签与 Ticket 关联管理

#### 3.2.1 为 Ticket 添加标签

- 用户可为某条 Ticket 添加一个或多个标签。
- 支持「输入即创建」：若输入的标签名在系统中不存在，则自动创建该标签再进行关联。
- 同一 Ticket 下，同一标签只能关联一次，重复关联需给出提示并忽略。
- 标签名长度限制：1–50 个字符，不可为空白字符。

#### 3.2.2 从 Ticket 移除标签

- 用户可从某条 Ticket 上移除指定标签（仅解除关联，不删除标签本身）。
- 移除操作即时生效，列表/详情中标签展示同步更新。

---

### 3.3 标签筛选

- 列表页提供标签筛选区域，展示所有已有标签（可附带各标签下的 Ticket 数量）。
- 支持**单选**或**多选**标签筛选：
  - 单选某标签：仅展示带有该标签的 Ticket。
  - 多选多个标签：展示同时带有所有选中标签的 Ticket（AND 逻辑）。
- 提供「无标签」快捷筛选项：仅展示未关联任何标签的 Ticket。
- 提供「全部」选项（默认状态），不按标签过滤。
- 标签筛选可与完成状态筛选、标题搜索联合使用。

---

### 3.4 完成状态筛选

- 列表页顶部提供完成状态筛选，选项为：
  - 全部（默认）
  - 未完成
  - 已完成
- 与标签筛选、标题搜索联合生效。

---

### 3.5 Ticket 颜色

- 每条 Ticket 可选择一种颜色作为视觉标识，用于在列表中快速区分不同类型的任务。
- 颜色从**预设调色板**中选取，共提供以下 8 种颜色（含「无颜色」默认态）：

| 颜色名 | 色值（参考） | 语义建议 |
|--------|-------------|---------|
| 无颜色 | — | 默认，无特殊标识 |
| 红色 | `#EF4444` | 紧急、阻塞、Bug |
| 橙色 | `#F97316` | 高优先级、待处理 |
| 黄色 | `#EAB308` | 注意、待确认 |
| 绿色 | `#22C55E` | 进行中、正常 |
| 蓝色 | `#3B82F6` | 功能、需求 |
| 紫色 | `#A855F7` | 改进、优化 |
| 灰色 | `#6B7280` | 低优先级、参考 |

- 颜色仅为视觉辅助，不参与筛选逻辑（本版本不支持按颜色过滤）。
- 颜色在 Ticket 卡片/行的**左侧边框**（4px 宽彩色竖线）上呈现；无颜色时边框透明或使用默认灰色边框。
- 颜色数据以颜色名称（英文枚举值）存储，前端负责将枚举值映射为对应的 CSS 色值。

---

### 3.6 标题搜索

- 列表页顶部提供搜索框，用户输入关键词对 Ticket 标题进行搜索。
- 搜索逻辑：不区分大小写的子串匹配（ILIKE 或等价实现）。
- 可选扩展：同时匹配描述（description）字段，由后端参数控制。
- 搜索与标签筛选、完成状态筛选联合生效（即结果为三者条件的交集）。
- 关键词为空时，视为不过滤。

---

## 4. 非功能需求

### 4.1 可用性

- 界面风格统一，使用 Shadcn UI 组件与 Tailwind CSS，保持响应式布局。
- 所有写操作（创建、编辑、删除、状态切换、标签增删）均提供明确的操作反馈：
  - 成功：Toast/消息提示或 UI 即时更新。
  - 失败：显示具体错误信息（校验错误或网络错误）。
- 关键破坏性操作（删除 Ticket）需二次确认。

### 4.2 数据一致性

- 所有写操作均通过后端 API 完成，不允许前端直接操作数据库。
- 标签名全局唯一，同名标签视为同一标签，不允许重复创建。
- Ticket 标题不可为空，后端须进行二次校验，不仅依赖前端校验。

### 4.3 性能

- 在 Ticket 数量 ≤ 1,000、标签数量 ≤ 100 时，列表加载与搜索响应时间应 < 1 秒。
- 列表支持分页（默认每页 20–50 条），防止大数据量下性能下降。
- 数据库对常用查询字段（title、completed、ticket_tag 外键）建立索引。

### 4.4 安全与配置

- 无用户系统，不做认证；若部署到非本机环境，需通过网络层（内网/反向代理）进行防护。
- 数据库连接字符串、API Base URL 等敏感配置通过环境变量注入，不硬编码在源码中。
- 后端配置 CORS，仅允许前端来源访问。

## 5. 数据模型设计

### 5.1 tickets 表

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | UUID | 主键，默认 gen_random_uuid() | 唯一标识 |
| title | VARCHAR(200) | NOT NULL | 标题 |
| description | TEXT | 可空 | 描述（纯文本） |
| color | VARCHAR(20) | 可空 | 颜色枚举值（red/orange/yellow/green/blue/purple/gray），null 表示无颜色 |
| completed | BOOLEAN | NOT NULL，默认 false | 完成状态 |
| created_at | TIMESTAMPTZ | NOT NULL，默认 now() | 创建时间 |
| updated_at | TIMESTAMPTZ | NOT NULL，默认 now() | 最后更新时间 |

颜色合法枚举值：`red`、`orange`、`yellow`、`green`、`blue`、`purple`、`gray`，以及 `null`（无颜色）。后端须校验传入值在枚举范围内，否则返回 400。

索引：
- `idx_tickets_title`：对 title 列建 GIN 或 BTREE 索引，支持模糊搜索。
- `idx_tickets_completed`：对 completed 列建索引，支持状态筛选。

### 5.2 tags 表

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | UUID | 主键，默认 gen_random_uuid() | 唯一标识 |
| name | VARCHAR(50) | NOT NULL，UNIQUE | 标签名（全局唯一） |
| created_at | TIMESTAMPTZ | NOT NULL，默认 now() | 创建时间 |

索引：
- `tags.name` 列已有 UNIQUE 约束，自动创建唯一索引。

### 5.3 ticket_tags 关联表（多对多）

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| ticket_id | UUID | NOT NULL，外键→tickets.id ON DELETE CASCADE | 关联的 Ticket |
| tag_id | UUID | NOT NULL，外键→tags.id ON DELETE CASCADE | 关联的 Tag |

约束：
- 联合主键 (ticket_id, tag_id)，防止重复关联。
- 删除 Ticket 时，对应的 ticket_tags 记录级联删除。
- 删除 Tag 时，对应的 ticket_tags 记录级联删除（解除关联）。

### 5.4 实体关系图（文字描述）

```
tickets (1) ─── (N) ticket_tags (N) ─── (1) tags
```

## 6. API 设计

### 6.1 响应格式约定

成功响应：HTTP 2xx，Body 为资源对象或列表对象。

```json
// 单资源
{ "id": "...", "title": "...", ... }

// 列表
{
  "items": [ {...}, {...} ],
  "total": 42,
  "limit": 20,
  "offset": 0
}
```

失败响应：HTTP 4xx/5xx，Body 统一含 `detail` 字段。

```json
{ "detail": "标题不能为空" }
```

---

### 6.2 Ticket 相关接口

#### GET /tickets

查询 Ticket 列表，支持筛选与搜索。

**Query 参数：**

| 参数 | 类型 | 说明 |
|------|------|------|
| q | string（可选） | 标题关键词（子串搜索，不区分大小写） |
| completed | boolean（可选） | true=已完成，false=未完成，不传=全部 |
| tag_ids | UUID[]（可选） | 标签 ID 列表，AND 逻辑，可重复传参 |
| no_tags | boolean（可选） | true=仅返回无标签 Ticket |
| limit | int（可选） | 每页数量，默认 20，最大 100 |
| offset | int（可选） | 分页偏移，默认 0 |

**响应：** 列表对象，每条 Ticket 含其关联标签列表。

---

#### POST /tickets

创建新 Ticket。

**请求 Body：**

```json
{
  "title": "修复登录页面的样式问题",
  "description": "在移动端 Safari 中按钮错位",
  "color": "red",
  "tag_ids": ["uuid-1", "uuid-2"]
}
```

`color` 可选，合法值为 `red`/`orange`/`yellow`/`green`/`blue`/`purple`/`gray`，不传或传 `null` 表示无颜色。

**响应：** HTTP 201，返回新建的 Ticket 对象（含 id、color、created_at 等）。

---

#### GET /tickets/{ticket_id}

获取单条 Ticket 详情，含关联标签。

**响应：** Ticket 对象。

---

#### PATCH /tickets/{ticket_id}

更新 Ticket 的标题、描述或完成状态（部分更新）。

**请求 Body（均为可选字段，至少一个）：**

```json
{
  "title": "新标题",
  "description": "新描述",
  "color": "blue",
  "completed": true
}
```

`color` 传 `null` 表示清除颜色；字段不传则保持原值不变。

**响应：** HTTP 200，返回更新后的 Ticket 对象。

---

#### DELETE /tickets/{ticket_id}

删除 Ticket 及其所有标签关联。

**响应：** HTTP 204，无 Body。

---

### 6.3 标签相关接口

#### GET /tags

获取所有标签列表，可选返回每个标签下的 Ticket 数量。

**Query 参数：**

| 参数 | 类型 | 说明 |
|------|------|------|
| with_count | boolean（可选） | true=附带各标签的 Ticket 数量 |

**响应示例：**

```json
[
  { "id": "uuid-1", "name": "bug", "ticket_count": 5 },
  { "id": "uuid-2", "name": "feature", "ticket_count": 12 }
]
```

---

#### POST /tags

创建新标签。

**请求 Body：**

```json
{ "name": "bug" }
```

**响应：** HTTP 201，返回新建标签对象。若同名标签已存在，返回 HTTP 409 并提示冲突。

---

#### DELETE /tags/{tag_id}

删除标签，同时级联删除所有关联的 ticket_tags 记录。

**响应：** HTTP 204。

---

### 6.4 Ticket 标签关联接口

#### POST /tickets/{ticket_id}/tags

为指定 Ticket 添加标签。

**请求 Body（二选一）：**

```json
// 方式 A：使用已有标签 ID
{ "tag_id": "uuid-1" }

// 方式 B：使用标签名（若不存在则自动创建）
{ "tag_name": "新标签" }
```

**响应：** HTTP 200，返回更新后的 Ticket 标签列表。若已关联，返回 HTTP 409。

---

#### DELETE /tickets/{ticket_id}/tags/{tag_id}

从指定 Ticket 移除标签（仅解除关联，不删除标签）。

**响应：** HTTP 204。

## 7. 前端页面与交互设计

### 7.1 整体布局

```
┌─────────────────────────────────────────────────────┐
│  顶部导航栏：Logo / 应用名称           [新建 Ticket] │
├──────────────┬──────────────────────────────────────┤
│              │  搜索栏 + 状态筛选                    │
│  标签侧边栏  ├──────────────────────────────────────┤
│              │  Ticket 列表                         │
│  · 全部      │                                      │
│  · 无标签    │  [Ticket 卡片 × N]                   │
│  · bug       │                                      │
│  · feature   │  [分页控件]                           │
│  · ...       │                                      │
└──────────────┴──────────────────────────────────────┘
```

### 7.2 主列表页

- **搜索栏**：顶部输入框，用户输入后防抖（300ms）触发搜索请求。
- **状态筛选**：Tab 或 SegmentedControl，选项为「全部 / 未完成 / 已完成」。
- **标签侧边栏**：
  - 列出所有标签，附带 Ticket 数量徽章。
  - 点击单个标签进行单选筛选；按住 Ctrl/Cmd 或提供多选模式进行多选筛选。
  - 提供「全部」与「无标签」两个固定选项。
- **Ticket 列表**：
  - 每条 Ticket 展示：彩色左边框（4px，对应 color 字段，无颜色时为透明/灰色）、完成状态切换（复选框）、标题、描述摘要（若有）、标签徽章列表、操作按钮组（编辑、管理标签、删除）。
  - 已完成 Ticket 标题添加删除线，行背景颜色轻微变化以示区别；颜色左边框仍保留但可降低饱和度。
  - 列表按 created_at 倒序排列（最新在上）。
- **分页控件**：底部分页，展示当前页码与总数。

### 7.3 创建 / 编辑 Ticket（模态对话框或侧边抽屉）

- **触发方式**：点击顶部「新建 Ticket」按钮打开创建表单；点击列表「编辑」打开编辑表单。
- **表单字段**：
  - 标题（必填，输入框，实时校验）
  - 描述（可选，Textarea）
  - 颜色（可选，调色板选择器，展示 8 个圆形色块供点击，含「无颜色」空白选项）
  - 标签（创建时可选，支持多选已有标签或输入新标签名）
- **操作按钮**：保存（主按钮）、取消（次按钮）。
- **校验错误**：在对应输入框下方显示错误提示文字；保存按钮在校验不通过时禁用或提示。

### 7.4 标签管理（Ticket 行内）

- 每条 Ticket 提供「管理标签」入口（图标按钮或标签区域点击）。
- 弹出 Popover 或 Dialog，展示：
  - 当前已关联标签（每个标签旁有删除图标）。
  - 标签输入/选择框：Combobox 形式，可搜索已有标签，也可输入新标签名按回车创建并关联。

### 7.5 删除 Ticket（确认对话框）

- 点击「删除」按钮后弹出 AlertDialog。
- 提示文案：「删除后无法恢复，确认要删除这条 Ticket 吗？」
- 操作按钮：确认删除（破坏性样式，红色）、取消。

## 9. 开发与部署

### 9.1 开发环境启动

- **数据库**：使用本地已安装的 PostgreSQL，不使用 Docker。数据库须提前创建（见 9.3）。
- **后端**：`uvicorn app.main:app --reload --port 8000`，通过 `.env` 配置数据库连接字符串。
- **前端**：`yarn dev`，通过 `.env` 配置 `VITE_API_BASE_URL=http://localhost:8000`。依赖安装使用 `yarn install`（或 `yarn`）。

### 9.2 环境变量说明

**后端 `.env`：**

| 变量名 | 实际值 | 说明 |
|--------|--------|------|
| DATABASE_URL | `postgresql://postgres:postgres@localhost:5432/projectalpha` | 数据库连接字符串 |
| ALLOWED_ORIGINS | `http://localhost:5173` | CORS 允许来源 |

**前端 `.env`：**

| 变量名 | 实际值 | 说明 |
|--------|--------|------|
| VITE_API_BASE_URL | `http://localhost:8000` | 后端 API 地址 |

### 9.3 数据库初始化

- 确认本地 PostgreSQL 已运行，使用以下命令创建数据库（仅首次）：

```sql
CREATE DATABASE projectalpha;
```

- 数据库连接信息：
  - 主机：`localhost`
  - 端口：`5432`
  - 数据库名：`projectalpha`
  - 用户名：`postgres`
  - 密码：`postgres`

- 使用 Alembic 执行迁移，创建所有表结构：

```bash
alembic upgrade head
```

## 10. 测试策略

### 10.1 后端测试

#### 10.1.1 测试框架与工具

| 工具 | 用途 |
|------|------|
| pytest | 测试运行器，主框架 |
| httpx + pytest-asyncio | 异步 HTTP 测试客户端，配合 FastAPI TestClient |
| pytest-cov | 测试覆盖率统计 |
| factory_boy 或 pytest fixtures | 测试数据构造 |

测试数据库使用独立的 PostgreSQL 数据库（如 `projectalpha_test`），测试前后通过 fixtures 自动创建/清理表数据，不污染开发库。

#### 10.1.2 单元测试（Unit Tests）

对独立的业务逻辑与工具函数进行测试，不依赖数据库或 HTTP 层：

- **数据校验**：Pydantic Schema 校验逻辑（标题为空、标题超长、颜色枚举非法值等）。
- **CRUD 函数**：对数据库操作函数（`crud/ticket.py`、`crud/tag.py`）进行测试，使用测试数据库执行真实 SQL。

#### 10.1.3 集成测试（Integration Tests）

通过 FastAPI `TestClient` 发起 HTTP 请求，测试完整的请求→路由→业务逻辑→数据库→响应链路：

**Ticket 相关：**

| 测试用例 | 预期结果 |
|----------|---------|
| POST /tickets，title 合法 | HTTP 201，返回含 id 的 Ticket 对象 |
| POST /tickets，title 为空 | HTTP 422，返回校验错误 |
| POST /tickets，color 非法值 | HTTP 400，返回错误提示 |
| GET /tickets，无筛选条件 | HTTP 200，返回列表对象 |
| GET /tickets?completed=true | 仅返回已完成 Ticket |
| GET /tickets?q=关键词 | 仅返回标题含关键词的 Ticket（不区分大小写） |
| GET /tickets?tag_ids=uuid1&tag_ids=uuid2 | 返回同时含两个标签的 Ticket（AND 逻辑） |
| GET /tickets?no_tags=true | 仅返回无标签 Ticket |
| PATCH /tickets/{id}，更新 title | HTTP 200，返回更新后对象，updated_at 刷新 |
| PATCH /tickets/{id}，color 置为 null | HTTP 200，color 字段为 null |
| DELETE /tickets/{id} | HTTP 204，再次 GET 该 id 返回 404 |
| DELETE /tickets/{id}，级联删除 ticket_tags | ticket_tags 表中对应记录已清除 |

**标签相关：**

| 测试用例 | 预期结果 |
|----------|---------|
| POST /tags，name 合法 | HTTP 201，返回新标签 |
| POST /tags，name 重复 | HTTP 409，返回冲突提示 |
| POST /tickets/{id}/tags，tag_name 不存在时 | 自动创建标签并关联，HTTP 200 |
| POST /tickets/{id}/tags，重复关联 | HTTP 409 |
| DELETE /tickets/{id}/tags/{tag_id} | HTTP 204，关联解除，标签本身仍存在 |
| DELETE /tags/{id} | HTTP 204，ticket_tags 中关联记录级联删除 |

#### 10.1.4 测试目录结构

```
backend/
└── tests/
    ├── conftest.py          # 测试数据库 Session、TestClient fixtures
    ├── test_tickets.py      # Ticket CRUD 集成测试
    ├── test_tags.py         # Tag CRUD 集成测试
    ├── test_ticket_tags.py  # 标签关联集成测试
    └── test_schemas.py      # Pydantic Schema 单元测试
```

#### 10.1.5 运行命令

```bash

pytest

pytest --cov=app --cov-report=term-missing
```
