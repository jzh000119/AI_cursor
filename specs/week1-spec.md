# Week1 项目：基于标签的 Ticket 管理工具 — 需求与设计文档

## 1. 项目概述

### 1.1 目标

构建一个轻量级、基于标签的 Ticket 管理工具，用于创建、分类、检索与完成工作任务。系统不包含用户账户与权限，面向单用户或本地团队使用。

### 1.2 技术栈

| 层级     | 技术选型                    |
|----------|-----------------------------|
| 数据库   | PostgreSQL                  |
| 后端     | FastAPI                     |
| 前端     | TypeScript + Vite + Tailwind CSS + Shadcn UI |

### 1.3 术语说明

- **Ticket**：一条待办/任务条目，包含标题、状态、可选描述等。
- **标签（Tag）**：用于分类的标记，一个 ticket 可关联多个标签，一个标签可关联多个 ticket（多对多）。

---

## 2. 功能需求

### 2.1 Ticket 管理

#### 2.1.1 创建 Ticket

- 用户可新建一条 ticket。
- **必填**：标题（title）。
- **可选**：描述（description）、初始关联的标签。
- 创建时默认状态为「未完成」。
- 需校验标题非空且长度在合理范围内（如 1–200 字符）。

#### 2.1.2 编辑 Ticket

- 用户可修改已有 ticket 的标题、描述。
- 不可直接在此处增删标签（通过「2.2 标签管理」完成）。
- 需校验修改后标题仍非空且长度合法。

#### 2.1.3 删除 Ticket

- 用户可删除一条 ticket。
- 删除后，该 ticket 与所有标签的关联一并删除（或由数据库级联处理）。
- 可选：删除前二次确认（前端或接口约定）。

#### 2.1.4 完成 / 未完成 Ticket

- 每条 ticket 具备状态：未完成、已完成。
- 用户可将 ticket 标记为「已完成」或改回「未完成」。
- 列表展示时需能区分两种状态（样式或筛选）。

### 2.2 标签与 Ticket 关联

#### 2.2.1 为 Ticket 添加标签

- 用户可为某条 ticket 添加一个或多个标签。
- 若标签尚不存在，可先创建该标签再关联，或支持「输入即创建」。
- 同一 ticket 下同一标签只能添加一次。

#### 2.2.2 删除 Ticket 的标签

- 用户可从某条 ticket 上移除指定标签。
- 仅解除关联，不删除标签本身（标签可被其他 ticket 使用）。

### 2.3 按标签查看 Ticket 列表

- 用户可按「全部」「未完成」「已完成」等状态筛选。
- 用户可按一个或多个标签筛选：仅展示带有所选标签的 ticket。
- 支持「无标签」视图：仅展示未关联任何标签的 ticket。
- 列表需支持分页或虚拟滚动（数据量较大时），具体策略在实现阶段定。

### 2.4 按标题搜索 Ticket

- 提供搜索框，用户输入关键词（如 title 的子串）。
- 搜索范围：ticket 的 title（可选：description）。
- 搜索可与上述状态、标签筛选组合使用。
- 匹配方式：不区分大小写的子串匹配（或前端/后端约定一致）。

## 3. 非功能需求

### 3.1 可用性与界面

- 前端使用 Shadcn UI 与 Tailwind，保持风格统一、响应式。
- 交互清晰：创建/编辑/删除有明确反馈（成功、失败、校验错误）。
- 关键操作（如删除）可做二次确认。

### 3.2 数据与一致性

- 所有写操作通过后端 API 完成，保证数据落在 PostgreSQL 中。
- 标签名可约定唯一（同一名称视为同一标签），避免重复创建。

### 3.3 性能与规模

- 在 ticket 与标签数量为百～千级时，列表加载与搜索响应时间应在可接受范围内（如 < 1s）。
- 后续可通过索引、分页、前端虚拟列表等做优化。

### 3.4 安全与部署

- 无用户系统，可不做认证；若部署到非本机，需考虑简单防护（如内网、反向代理、环境变量配置等）。
- 数据库连接信息、API 地址等通过配置或环境变量管理，不写死在前端代码中。

## 4. 技术架构概览

### 4.1 整体结构

```
[ 浏览器 ]  <--->  [ FastAPI 后端 ]  <--->  [ PostgreSQL ]
     |                      |
   Vite + TS              REST API
   Tailwind
   Shadcn UI
```

### 4.2 后端职责

- 提供 RESTful API：ticket 与标签的 CRUD、关联的增删、列表筛选与搜索。
- 校验请求参数，返回统一格式的 JSON（含成功/失败与错误信息）。
- 使用 ORM（如 SQLAlchemy）或直接 SQL 访问 PostgreSQL。

### 4.3 前端职责

- 单页应用（SPA）：列表页、创建/编辑表单、搜索与筛选、标签管理入口。
- 通过 HTTP 客户端（如 fetch 或 axios）调用后端 API，不直接连数据库。
- 状态管理：可用 React state、Context 或轻量 store，保证列表与表单与后端一致。

### 4.4 数据库

- 仅后端连接 PostgreSQL；需建表：tickets、tags、ticket_tag 关联表（见下节）。

## 5. 数据模型设计

### 5.1 表结构概念

#### 5.1.1 tickets 表

| 字段名       | 类型         | 说明                     |
|--------------|--------------|--------------------------|
| id           | 主键（如 UUID 或自增） | 唯一标识                 |
| title        | 字符串       | 标题，非空               |
| description  | 文本（可空） | 描述                     |
| completed    | 布尔         | 是否已完成，默认 false   |
| created_at   | 时间戳       | 创建时间                 |
| updated_at   | 时间戳       | 最后更新时间             |

#### 5.1.2 tags 表

| 字段名     | 类型         | 说明           |
|------------|--------------|----------------|
| id         | 主键         | 唯一标识       |
| name       | 字符串，唯一 | 标签名称       |
| created_at | 时间戳       | 创建时间（可选） |

#### 5.1.3 ticket_tag 关联表（多对多）

| 字段名    | 类型 | 说明                    |
|-----------|------|-------------------------|
| ticket_id | 外键 | 引用 tickets.id         |
| tag_id    | 外键 | 引用 tags.id            |
| 主键      | (ticket_id, tag_id) | 联合主键，防止重复关联 |

### 5.2 关系说明

- 一个 ticket 可对应多条 ticket_tag，即多个 tag。
- 一个 tag 可对应多条 ticket_tag，即多个 ticket。
- 删除 ticket 时，可级联删除其 ticket_tag 记录；删除 tag 时，可级联删除 ticket_tag，或禁止删除仍被引用的 tag，依产品偏好而定。

## 6. API 设计概要

### 6.1 Ticket 相关

- `GET /tickets` — 列表与筛选：查询参数可含 `completed`、`tag_ids[]`、`q`（搜索）、`limit`、`offset`。
- `POST /tickets` — 创建：body 含 title、可选 description、可选 tag_ids。
- `GET /tickets/{id}` — 单条详情（含其标签）。
- `PATCH /tickets/{id}` — 更新 title、description、completed。
- `DELETE /tickets/{id}` — 删除。

### 6.2 标签与关联

- `GET /tags` — 标签列表（可选：带每个 tag 的 ticket 数量）。
- `POST /tags` — 创建标签（name 唯一）。
- `POST /tickets/{id}/tags` — 为 ticket 添加标签（body 含 tag_id 或 name）。
- `DELETE /tickets/{id}/tags/{tag_id}` — 移除 ticket 的某标签。
- 可选：`DELETE /tags/{id}` — 删除标签（若存在关联可拒绝或级联，需在设计中明确）。

### 6.3 响应格式约定

- 成功：HTTP 2xx，body 为资源对象或列表。
- 失败：HTTP 4xx/5xx，body 含 `detail` 或 `message` 及可选错误码，便于前端统一展示。

## 7. 前端页面与交互设计

### 7.1 页面结构建议

- **主列表页**
  - 顶部：标题搜索框、状态筛选（全部/未完成/已完成）、标签筛选（多选或下拉）。
  - 列表：每条 ticket 展示标题、完成状态、标签列表、操作按钮（编辑、删除、切换完成、管理标签）。
  - 支持「按标题搜索」与当前筛选组合。

- **创建 / 编辑 Ticket**
  - 表单：标题（必填）、描述（可选）、标签选择/输入（多选或标签输入框）。
  - 提交后刷新列表或就地更新；校验错误在表单旁或顶部提示。

- **标签管理入口**
  - 可在列表行内「添加/删除标签」弹窗或侧栏完成；
  - 或单独一个「标签管理」页，管理全局标签列表（可选）。

### 7.2 交互要点

- 列表与筛选、搜索联动，无刷新或局部刷新均可，以实现简单、体验清晰为准。
- 完成/未完成可用复选框或按钮，即时请求 API 并更新 UI。
- 删除前确认、网络错误与校验错误的提示需明确。

### 7.3 UI 组件建议

- 使用 Shadcn 的 Table、Card、Button、Input、Dialog、Badge（标签展示）、Select 或 Combobox（标签选择）等，保持风格一致。

## 8. 部署与运行环境

### 8.1 开发环境

- 本地安装 PostgreSQL，或使用 Docker 运行 Postgres。
- 后端：FastAPI 运行于本地端口（如 8000），CORS 允许前端来源。
- 前端：Vite 开发服务器（如 5173），通过环境变量配置 API 基地址。

### 8.2 生产或演示部署（可选）

- 后端可部署于任意支持 Python 的服务器或容器；数据库使用同一 Postgres 或托管型 Postgres。
- 前端构建为静态资源，由 Nginx 或其他 Web 服务器托管，或与后端同域提供。
- 若前后端分离部署，需配置 CORS 与 API 基地址。
